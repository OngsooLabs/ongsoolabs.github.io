OSL 통합 개발 가이드라인 v1.1

1. 네임스페이스(Namespace) 표준화

모듈 간 충돌을 방지하고 코드 자동 완성을 쾌적하게 만들기 위해, 모든 프로젝트의 루트 네임스페이스는 OSL로 시작합니다. 네임스페이스 뎁스를 간결하게 유지합니다.

  • 패턴: OSL.[ProjectName].[Module]

[기존 프로젝트 마이그레이션 맵]

  1. EasyPatcher: OSL.EasyPatcher.Runtime, OSL.EasyPatcher.Editor
  2. SecureSave: OSL.SecureSave.Runtime, OSL.SecureSave.Editor
  3. IapClient: OSL.IapClient.Runtime, OSL.IapClient.Editor
  4. SocialSignin - Core: OSL.SocialLogin.Core
  5. SocialSignin - Facebook: OSL.SocialLogin.Facebook
  6. SocialSignin - Google: OSL.SocialLogin.Google
  7. SocialSignin - Apple: OSL.SocialLogin.Apple
  8. SocialSignin - Steam: OSL.SocialLogin.Steam
  9. (신규) 공통 유틸리티: OSL.Core

2. 폴더 구조(Folder Structure) 표준화

에셋 스토어 검수를 통과하고 유저(개발자) 환경에서의 가시성을 확보하기 위해 아래의 디렉토리 구조를 강제합니다. 그 외 불필요한 Depth는 생성하지 않습니다.

Assets/OSL/
  ├── Core/                       # 공통 유틸리티 (로깅, 암호화 등)
  └── [ProjectName]/              # 개별 에셋 패키지 (예: EasyPatcher)
       ├── package.json           # UPM 호환 및 버전 명시
       ├── Runtime/               # 메인 게임 로직 및 스크립트 (.asmdef)
       ├── Editor/                # 에디터 윈도우, 인스펙터, 빌드 로직 (.Editor.asmdef)
       ├── Plugins/               # 외부 SDK AAR, DLL, iOS(.m/.mm) 파일 격리
       ├── Samples/               # 유저가 바로 테스트 가능한 Demo Scene 및 Sample Script
       └── Documentation/         # README.pdf, README.md 등 문서 정리 공간

3. Assembly Definition (ASMDEF) 관리 원칙

모든 스크립트는 적절한 Assembly로 묶여 빠른 컴파일과 결합도 최소화를 보장해야 합니다.

  • 네이밍 규칙:
    • Runtime: OSL.[ProjectName].asmdef
    • Editor: OSL.[ProjectName].Editor.asmdef
  • 성능 최적화 스펙:
    • Auto Referenced: False 설정 강제 (모든 의존성은 명시적 참조 필수)
    • Use GUIDs: True (에셋 폴더 이동 시 레퍼런스 깨짐 방지)
  • 외부 라이브러리(Facebook SDK, Steamworks 등) 처리 방안:
    • 사용자의 프로젝트에 외부 SDK가 설치되지 않은 상태에서 OSL 파일을 임포트해도 컴파일 에러가 발생하지 않아야 합니다.
    • Version Defines (Define Constraints) 필수 활용:
      • 예: Steamworks 패키지가 감지될 때만 OSL_HAS_STEAM define을 활성화.
      • 코드 내부에서는 #if OSL_HAS_STEAM ... #endif로 분기 처리하여, 유저가 SDK를 포함하기 전까지는 코드가 컴파일에서 제외되도록(Soft Dependency) 구성.

4. 계층형 Core 아키텍처 설계

고객의 프로젝트에 불필요한 Bloatware(비대화된 코드)가 들어가지 않도록 계층적 샌드박스를 구성합니다. 상위 모듈은 하위 모듈만 참조할 수 있습니다 (단방향 의존성).

Layer 0: OSL.Core (Shared Utility)

  • 역할: 어떤 에셋이든 사용하는 초경량 공용 모음집.
  • 기능: OSLCrypto.cs (AES/HMAC), OSLLogger.cs (공통 로그 인터페이스), 네트워크 웹베이스 통신 모듈.
  • 의존성: 유니티 기본 기능 외 외부 모듈 의존 불가.

Layer 1: OSL.Auth.Base (Domain Core)

  • 역할: 소셜 로그인 모듈 5종을 묶어주는 인증 베이스.
  • 기능: ISignin 인터페이스, 토큰 검증, 로그인 상태 관리.
  • 의존성: OSL.Core 참조.

Layer 2: OSL.Product (End Product)

  • 역할: 단일 목적을 수행하는 배포 단위.
  • 예시: OSL.EasyPatcher, OSL.SocialLogin.Steam, OSL.SecureSave
  • 의존성: 플러그인이 동작하기 위한 최소 Layer(Core 또는 Auth.Base)만 선별적으로 참조.

5. 버전 및 배포(.unitypackage) 자동화 준비 체계

5-1. 버전 관리 (package.json)

모든 에셋 폴더에는 유니티 패키지 매니저(UPM)와 호환되는 package.json을 작성해야 합니다.

{
  "name": "com.osl.easypatcher",
  "version": "1.0.0",
  "displayName": "OSL Easy Patcher",
  "dependencies": {
    "com.osl.core": "1.0.0" 
  }
}

5-2. 스토어 배포용 추출(Export) 체크리스트 및 자동화 원칙

기존 모노레포 구조에서 개별 에셋을 분리하여 에셋 스토어로 내보낼 때, 다음 체크리스트를 준수합니다 (향후 Editor 스크립트로 Export 자동화 권장).

  1. 의존성 자동 포함 검증:
    • OSL.SocialLogin.Steam 추출 시 의존하고 있는 OSL.Auth.Base와 OSL.Core 폴더가 .unitypackage에 자동 포함 체크 되었는지 확인.
  2. Samples 격리 확인:
    • 샘플 씬 안의 임시 텍스처, 테스트용 프리팹 코드가 Runtime 폴더를 오염시키지 않았는지 검증.
  3. Third-Party SDK 배제:
    • 에셋 스토어 규정상 타사의 SDK(Facebook SDK 등)를 재배포할 수 없으므로, 반드시 타사 플러그인은 Export 항목에서 체크 해제하여 내보냅니다. 유저는 OSL 가이드 문서의 링크를 통해 직접 SDK를 다운받도록 유도합니다.
  4. 민감 데이터 검열:
    • 테스트용 aws_credentials, google-services.json, API Key 등이 남아있는지 검사 및 배제.
  5. 문서(Docs) 검수:
    • 해당 에셋 디렉토리 내 Documentation/README.pdf 파일 유무 및 영어/다국어 지원 여부 확인(사용자 기본 설정).
