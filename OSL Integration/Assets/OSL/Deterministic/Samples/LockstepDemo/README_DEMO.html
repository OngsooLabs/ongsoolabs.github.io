OSL Deterministic Core — Lockstep Demo

This sample is a visual proof that OSL Deterministic Core produces bit-for-bit identical results across independent instances when seeded with the same value. It is the most direct way to verify the asset's central promise before shipping it into a production project.

What it shows

Two completely independent LockstepSimulation instances are constructed from the same seed and ticked side by side. Each tick advances through:

  1. A 52-card Fisher–Yates shuffle (OSLDeckShuffler).
  2. A 10-pull weighted gacha session with hard pity (OSLGachaTable + SimpleHardPity).
  3. Five turns of dice/crit/dodge combat resolution (OSLCombatResolver).

After every tick the demo computes a 64-bit FNV-1a hash over the full simulation state (engine state, pity counter, phase, turn, HPs, deck top) for both panels and displays:

  • ● SYNC (green) when the hashes match.
  • ✗ DESYNC (red) the moment they diverge.

A regression in any deterministic module would cause the banner to turn red instantly.

Setup

  1. Ensure the OSL.Deterministic.Runtime assembly compiles cleanly in your project.
  2. Open any scene (a fresh empty scene is fine).
  3. Create an empty GameObject named LockstepDemo.
  4. Add the OSL > Deterministic > Lockstep Demo component to it.
  5. Press Play.

The demo runs entirely on OnGUI, so no Canvas, TextMeshPro, or UI Toolkit dependencies are required.

Controls

Control Description
Auto: ON/OFF Toggles automatic ticking on the configured interval.
Step Advances both simulations by exactly one tick.
Reseed Rebuilds both simulations from the current seed value.
Seed (hex) 16-character hexadecimal seed. Both simulations use this value.

Why this matters

Lockstep multiplayer, replay systems, automated QA, and server-authoritative simulations all require strict determinism. OSL Deterministic Core was designed from the ground up around that constraint:

  • Zero third-party dependencies, .NET Standard 2.1 only.
  • Zero allocations on RNG hot paths; Span<byte> state save/load.
  • Integer-first probability APIs to avoid floating-point drift across platforms.
  • Pluggable IRandomEngine and IPitySystem contracts for full control.

If this demo runs without ever showing DESYNC, the runtime is delivering on that promise.
