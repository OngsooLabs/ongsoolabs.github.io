OSL IAP Client

Server-authoritative In-App Purchase verification for Unity.
Purchase receipts are validated by your own backend server before rewards are granted, preventing client-side receipt forgery.

Overview

OSL IAP Client wraps Unity IAP and adds a server-side verification step:

  1. Player initiates a purchase → Unity IAP processes payment with the store.
  2. On success, IapManager sends the receipt to your backend for verification.
  3. Backend validates with the store (Apple/Google) and returns the result.
  4. Only after a successful backend response are in-game rewards granted.

This prevents replay attacks and client-side receipt manipulation.

com.unity.purchasing 4.x / 5.x Unity IAP

Backend server — Endpoint to verify receipts (see _serverUrl in IapSetupConfig)

Dependency Notes
com.unity.inputsystem If your project uses the new Input System, see Input System Compatibility

If com.unity.ugui or com.unity.purchasing are missing, the package still imports successfully. Runtime scripts enter a compatibility stub mode and log a warning until the required packages are installed.

  • OSL_IAP_HAS_UGUI — defined automatically when com.unity.ugui is detected. Controls whether the full IapDemoUI compiles or a no-op stub is used.
  • OSL_IAP_HAS_PURCHASING — defined automatically when com.unity.purchasing is detected. Controls IapManager's purchase flow.

Installation

  1. Open the Unity Asset Store (OSL IAP Client) and search for OSL IAP Client, or open it directly via your browser.
  2. Click Add to My Assets.
  3. Open Unity and navigate to Window > Package Manager.
  4. Select Packages: My Assets from the dropdown.
  5. Locate OSL IAP Client, download it, and click Import.
  6. The scripting define symbols mentioned above are set automatically on import.
  7. Open Assets/OSL/IapClient/Resources/IapSetupConfig.asset and fill in your server URL and product IDs.

Quick access via menu: Tools → OSL → IAP API Tester → Create Setup Asset

Backend Endpoint

Your server must expose a POST endpoint that receives the receipt payload from BackendApiClient and returns a JSON response indicating success or failure. See Backend Configuration in this document for the full server-side setup.

Scene Setup

  1. Add IapManager and BackendApiClient as components on a GameObject.
  2. Assign the same IapSetupConfig asset to both components.
  3. Wire BackendApiClient to IapManager._apiClient.

Demo Scene

Located at Assets/OSL/IapClient/Samples/DemoScene.unity.

The demo scene includes:

  • Login Panel — enter a User ID to simulate a logged-in player.
  • Gold Panel — triggers a consumable IAP (gold_100 by default).
  • VIP Panel — triggers a subscription IAP (vip_monthly by default).
  • Log Panel — scrollable log of all events and results.

Button Wiring Warning
Buttons are wired automatically at runtime via AddListener in Start().
Do not assign OnClickLogin / OnClickBuyGold / OnClickBuyVIP through the Inspector's OnClick list. Doing so will cause each button to fire twice.

Input System Compatibility

Background

The demo scene does not serialize an input module component in its YAML.
Instead, IapDemoUI ensures a valid input module exists at runtime in Start().

Automatic Detection

IapDemoUI detects the active input backend at startup and adds the appropriate module:

Active Input Handling (Player Settings) Module Added
Input Manager (Old) StandaloneInputModule
Input System Package (New) InputSystemUIInputModule (via reflection)
Both StandaloneInputModule

Detection works by calling UnityEngine.Input.GetButtonDown("Submit") inside a try/catch(InvalidOperationException). If the exception fires, the project is Input System only, and the code searches loaded assemblies for InputSystemUIInputModule instead.

Known Issue — InvalidOperationException on UnityEngine.Input

Symptom:

InvalidOperationException: You are trying to read Input using the UnityEngine.Input class,
but you have switched active Input handling to Input System package in Player Settings.
UnityEngine.EventSystems.StandaloneInputModule.ShouldActivateModule ()

Cause:
Active Input Handling is set to Input System Package (New) only, but a StandaloneInputModule is already present on the EventSystem GameObject (from a stale import or manual addition).

Fix:

  1. Remove the StandaloneInputModule component from the EventSystem in your scene.
  2. Ensure com.unity.inputsystem is installed.
  3. Re-enter Play mode — IapDemoUI will add InputSystemUIInputModule automatically.

If InputSystemUIInputModule cannot be found:

[OSL IAP] Active input handling is Input System, but InputSystemUIInputModule could not be found.
Add it to the EventSystem manually.

This means com.unity.inputsystem is not installed. Either install it, or change Active Input Handling back to Input Manager (Old) in Edit → Project Settings → Player → Other Settings.

Recommendation for Imported Projects

Check Edit → Project Settings → Player → Other Settings → Active Input Handling:

  • Input System Package (New) → install com.unity.inputsystem before running the demo scene.
  • Both or Input Manager (Old) → no additional steps needed.

Purchase Response Contract

The backend verification endpoint returns a PurchaseResponse with:

Field Values
ResultCode Verified | AlreadyProcessed | VerificationFailed
ShouldConfirmPurchase bool
ShouldGrantReward bool

Client must follow these rules exactly:

  1. If ShouldConfirmPurchase == true → call ConfirmPurchase on the store.
  2. If ShouldGrantReward == true → grant the item exactly once.
  3. If ResultCode == AlreadyProcessed → confirm the pending store order, but do not grant the reward again.

First-Success Test Path

  1. Open DemoScene.unity and assign IapSetupConfig (Server URL, product IDs, etc.).
  2. Enter Play mode.
  3. Enter a User ID and click Login.
  4. Click Buy Gold or Buy VIP to trigger a test purchase.
    • Apple: use a Sandbox Tester account; pass IsSandbox = true in the config.
    • Google Play: use the Play Store test purchase flow.
  5. Confirm the Console log shows the server verification result.
  6. Confirm the reward is granted exactly once.
  7. Confirm the store order is confirmed (ConfirmPurchase called).
  8. Repeat the same transaction and confirm no duplicate reward is granted.

Troubleshooting

Problem Solution
Buttons are not clickable Check that an input module is present on the EventSystem GameObject. See Input System Compatibility.
Missing script on EventSystem A stale serialized reference to an input module exists. Remove the unknown component from the EventSystem and let IapDemoUI add the correct one at runtime.
IapSetupConfig fields not saving Make sure you are editing the .asset file in Assets/OSL/IapClient/Resources/, not a copy.
Purchases succeed but rewards not granted Verify that _serverUrl in IapSetupConfig points to a reachable backend and that the endpoint returns the expected JSON response.
InvalidOperationException from StandaloneInputModule See Known Issue above.
OSL_IAP_HAS_UGUI not defined Ensure com.unity.ugui (or com.unity.modules.ui) is listed in your project's manifest.json.
OSL_IAP_HAS_PURCHASING not defined Ensure com.unity.purchasing is installed via Package Manager.

Backend Infrastructure Setup

The following sections cover the server-side configuration required for receipt verification and real-time webhook notifications.

Google Play Store Setup

To verify purchases and receive real-time subscription updates, you must link Google Cloud Platform (GCP) with your Google Play Console.

1.1 GCP Service Account

  1. Go to https://console.cloud.google.com/
  2. Create a new project (or select an existing one).
  3. Navigate to APIs & Services → Library and enable the Google Play Android Developer API.
  4. Go to IAM & Admin → Service Accounts and click Create Service Account.
  5. Name it (e.g. iap-backend-service). No role assignment needed here — permissions are granted through the Play Console.
  6. Click the created service account → Keys tab → Add Key → Create new key → JSON.
  7. Download the .json file and place it on the server, e.g. Keys/google-service-account.json.

1.2 Google Play Console — API Access

  1. Go to https://play.google.com/console
  2. Navigate to Setup → API access.
  3. Link the GCP project created in 1.1.
  4. Scroll to Service Accounts, find your account, click Manage Play Console permissions.
  5. Grant the following permissions:
    • View app information and download bulk reports (Read-only)
    • View financial data, orders, and cancellation survey responses
    • Manage orders and subscriptions

1.3 Google Pub/Sub Webhook (Real-Time Developer Notifications)

  1. In Google Cloud Console, go to Pub/Sub → Topics and create a topic (e.g. play-store-notifications).
  2. Add a permission to the topic:
    • Principal: google-play-developer-notifications@system.gserviceaccount.com
    • Role: Pub/Sub Publisher
  3. Go to Pub/Sub → Subscriptions and create a Push subscription:
    • Endpoint URL: https://your-domain.com/api/webhook/google
    • Authentication: Enable OIDC token
    • Audience: Must match the value in Iap:Google:PubSubAudience
    • Service Account: A dedicated push-invoker SA; its email goes in Iap:Google:PubSubServiceAccountEmail
  4. In Google Play Console, go to Monetize → Setup → Monetization setup.
  5. Under Real-time developer notifications, enter your Pub/Sub topic name and save.

The backend rejects Google webhook calls that do not include a valid Pub/Sub OIDC bearer token.
A plain unauthenticated push subscription will not work.

Apple App Store Setup

Apple uses the App Store Server API V2 for receipt verification and App Store Server Notifications V2 for webhooks.

2.1 Apple Developer Portal — Generating Keys

  1. Go to https://developer.apple.com/ and log in.
  2. Navigate to Certificates, Identifiers & Profiles → Keys.
  3. Click + to create a new key.
  4. Name it (e.g. OSL IAP Server Key) and check In-App Purchase.
  5. Click Continue → Register.
  6. Download the .p8 file. Warning: This file can only be downloaded once. Do not lose it.
  7. Place it on the server, e.g. Keys/AuthKey.p8.
  8. Note the Key ID (Keys page) and your Issuer ID (App Store Connect → Users and Access → Keys → App Store Connect API).

2.2 App Store Connect — App Setup

  1. Go to https://appstoreconnect.apple.com/
  2. Select your app under My Apps.
  3. Verify the Bundle ID matches your Unity project and appsettings.json.
  4. Set up In-App Purchases under Features → In-App Purchases.

2.3 Apple Webhook (App Store Server Notifications V2)

  1. In App Store Connect, go to your app → App Information (General section).
  2. Scroll to App Store Server Notifications.
  3. Enter your webhook endpoint in both Production and Sandbox fields: https://your-domain.com/api/webhook/apple
  4. Select Version 2 for the notification format and save.

2.4 Apple Root Certificate for JWS Verification

  1. Download the Apple Root CA certificate (AppleRootCA-G3.cer) from Apple's certificate authority page.
  2. Store it on the server, e.g. Keys/AppleRootCA-G3.cer.
  3. Set Iap:Apple:RootCertPath to that file path.

The backend rejects Apple webhook payloads unless the JWS certificate chain validates against this pinned root certificate.

Backend Configuration

Update appsettings.json on the server with the credentials gathered above:

{
  "ConnectionStrings": {
    "DefaultConnection": "Data Source=iap_database.db"
  },
  "Iap": {
    "Google": {
      "PackageName": "com.yourcompany.yourgame",
      "ServiceAccountKeyPath": "Keys/google-service-account.json",
      "PubSubAudience": "https://your-domain.com/api/webhook/google",
      "PubSubServiceAccountEmail": "pubsub-invoker@your-project.iam.gserviceaccount.com"
    },
    "Apple": {
      "BundleId": "com.yourcompany.yourgame",
      "IssuerId": "YOUR_APPLE_ISSUER_ID",
      "KeyId": "YOUR_APPLE_KEY_ID",
      "PrivateKeyPath": "Keys/AuthKey.p8",
      "RootCertPath": "Keys/AppleRootCA-G3.cer"
    }
  }
}

Feature-to-Configuration Mapping

Endpoint Required Configuration
GET /api/iap/history/{userId} Database connection only
GET /api/iap/subscription/{userId} Database connection only
POST /api/iap/google/verify Iap:Google:PackageName, Iap:Google:ServiceAccountKeyPath
POST /api/iap/apple/verify Iap:Apple:BundleId, IssuerId, KeyId, PrivateKeyPath, RootCertPath
POST /api/webhook/google Iap:Google:PubSubAudience, Iap:Google:PubSubServiceAccountEmail
POST /api/webhook/apple Iap:Apple:RootCertPath

The read-only history/subscription APIs do not require Google or Apple key files as long as the database is configured.

What Can Be Omitted (No Webhooks)

Commonly optional:

  • Iap:Google:PubSubAudience
  • Iap:Google:PubSubServiceAccountEmail

Google-only verification:

  • Required: Iap:Google:PackageName, Iap:Google:ServiceAccountKeyPath
  • Omittable: entire Apple section, all Pub/Sub fields

Apple-only verification:

  • Required: Iap:Apple:BundleId, IssuerId, KeyId, PrivateKeyPath, RootCertPath
  • Omittable: entire Google section, all Pub/Sub fields

ConnectionStrings:DefaultConnection is optional. Omitting it falls back to Data Source=iap_database.db.

Database Management

The backend uses SQLite by default for zero-configuration testing. It is built on Dapper (Micro-ORM) and can be scaled to MySQL or PostgreSQL with minimal changes.

Default SQLite Setup

  • No database server required.
  • On first run, InitializeDatabase() creates iap_database.db in the application root.
  • To view data, use DB Browser for SQLite or DBeaver.

Migrating to MySQL / PostgreSQL

  1. Install the provider: 
    dotnet add package MySql.Data    # MySQL
dotnet add package Npgsql         # PostgreSQL
  2. In SqlitePurchaseDao.cs, replace new SqliteConnection(...) with new MySqlConnection(...).
  3. Update ConnectionStrings:DefaultConnection in appsettings.json.

The SQL in this backend is standard ANSI SQL and is compatible with MySQL, PostgreSQL, and SQL Server with minimal or no changes.

Warnings & Pro Tips

General

  • HTTPS is mandatory. Both Google Pub/Sub and Apple Server Notifications require a valid SSL/TLS certificate. For local testing, use ngrok to expose localhost over HTTPS.
  • Idempotency: Webhooks may be delivered more than once. The backend handles this, but never remove the database checks that prevent duplicate processing.
  • Duplicate Purchase Handling: The verify API returns a semantic success for AlreadyProcessed. The client must confirm the store order but must not grant the reward again.

Google Play

  • Propagation Delay: Service Account permissions can take 24–48 hours to fully propagate after being granted in the Play Console. If you receive "Purchase not found" or 401 Unauthorized immediately after setup, wait and retry.
  • Testing Sandbox: Add test Google accounts to the License Testing section in the Play Console to test IAP without real charges.

Apple

  • OriginalTransactionId Rule: Apple generates a new TransactionId for every subscription renewal, but webhooks only send OriginalTransactionId. This backend uses OriginalTransactionId as the subscription Primary Key so renewals are tracked correctly.
  • Sandbox vs Production: When testing with a Sandbox Tester account via Unity/Xcode, the client must pass IsSandbox = true in the API request. Otherwise, the server verifies the test receipt against Apple's production servers, which will fail.
  • Sandbox Webhooks: Apple's sandbox simulates time differently (e.g. a 1-month subscription renews every 5 minutes). Use this to rapidly test DID_RENEW and EXPIRED webhook logic.
