OSL Social Login Pro

OSL Social Login Pro is a unified, production-ready security package that seamlessly integrates client-side social authentication (SocialLogin) with a robust backend token verification server boilerplate (TokenVerification).

By bridging client-side credential retrieval with backend cryptographic signature and ticket verification, this suite provides an end-to-end secure sign-in pipeline for Unity-based multiplayer games and applications.

Module Architecture

To maintain high modularity and clean decoupling, the package is organized into two primary sub-folders. No physical code migration is required, and they are compiled as independent modules.

  1. SocialLogin:
    • Manages client-side SDK integration and OAuth browser loops.
    • Retrieves identity tokens (Google ID Token, Apple JWT), access tokens (Facebook Access Token, Discord Auth Code), or session tickets (Steam Session Ticket).
  2. TokenVerification:
    • Provides a client-side verification manager (TokenVerifyManager) that communicates via HTTP POST with the verification backend.
    • Includes a standalone, optimized .NET Web API server boilerplate (TokenVerificationServer) to process cryptographically secure token validation.

Detailed Usage Guide

Step 1: Configure Client-Side Social Login

  1. Open the Unity Editor and navigate to Tools -> OSL -> Social Login Settings.
  2. Toggle on the social identity providers you wish to enable (e.g., Google, Facebook, Apple, Steam, Discord). This automatically configures platform-specific scripting define symbols (e.g., OSL_GOOGLE).
  3. Fill in your client credentials (such as Client IDs, App IDs, and Redirect URIs) into the generated SocialLoginSettings.asset under the Resources folder.

Step 2: Spin up the Verification Server

  1. Navigate to the server project folder (TokenVerificationServer located outside the Unity Assets folder).
  2. Open appsettings.json and enter your secure client secrets, developer tokens, or Web API keys:
    • Google: GoogleClientId
    • Facebook: FacebookAppId, FacebookAppSecret
    • Apple: AppleAppId (Bundle ID)
    • Steam: SteamAppId, SteamWebApiKey
    • Discord: DiscordClientId, DiscordClientSecret, DiscordRedirectUri
  3. Run the server using: 
    dotnet run --project OSL.TokenServer.csproj
    By default, the development environment hosts the API Swagger UI at http://localhost:5181/swagger.

Step 3: Link Authentication with Verification

Inside your game controller or authentication manager script, retrieve the token from the SocialLogin callback and pass it directly to TokenVerifyManager for secure validation:

// Example: Google Sign-In with backend token verification
SocialLoginManager.Instance.Login(ProviderKind.G, (success, result) =>
{
    if (success)
    {
        // forward retrieved ID Token to your backend server
        string idToken = result.Token;
        
        TokenVerifyManager verifyManager = GetComponent<TokenVerifyManager>();
        verifyManager.VerifyToken(AuthProvider.Google, idToken, (response) =>
        {
            if (response.success)
            {
                Debug.Log($"Verification Success! User Unique ID (UID): {response.uid}");
                // Proceed to game server login with the verified UID
            }
            else
            {
                Debug.LogError($"Verification Failed: {response.message}");
            }
        });
    }
    else
    {
        Debug.LogError("Social Login Failed.");
    }
});

Under-the-Hood Optimizations

The SocialLoginPro backend server implements several key optimizations and security measures:

  • Apple JWKS Caching: Apple's public signature keys (JWKS) are cached in-memory for 12 hours. This drastically reduces HTTP requests to Apple servers and prevents login API failures during momentary Apple outages.
  • Steam URL Safety: Outgoing requests to the Steamworks Web API utilize URL encoding (Uri.EscapeDataString) to protect session hex tickets containing special characters from transmission errors.
  • Discord Redirect Validation: Incoming authentication code exchanges are cross-referenced with the server-side configured Redirect URI to block redirect tampering vulnerabilities.
