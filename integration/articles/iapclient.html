IapClient

IapClient is a Unity package that integrates in-app purchase (IAP) functionality into your project.
It is built on top of Unity IAP v5 and Unity Services Core, and provides a server-side receipt verification workflow against your own backend.

Features

  • Google Play and Apple App Store purchase flow integration
  • Backend receipt verification with automatic retry
  • Duplicate reward grant prevention
  • Environment-specific configuration via the IapSetupConfig ScriptableObject
  • In-editor backend API test tool (IAP API Tester)

Prerequisites

Package Package ID
Unity Purchasing com.unity.purchasing
Unity Services Core com.unity.services.core

If either package is missing, IapManager compiles as a fallback stub and all runtime IAP functionality is disabled.

Installation

  1. Search for OSL IAP Client on the Unity Asset Store, or open it directly at Unity Asset Store (OSL IAP Client), then click Add to My Assets.
  2. In Unity, open Window > Package Manager, select Packages: My Assets, locate the package, and click Download then Import.
  3. Wait for Unity to finish script compilation.

Configuring IapSetupConfig

IapSetupConfig is a ScriptableObject that centralizes all IAP environment settings.
Create one via the Create Setup Asset button in the Tools > OSL > IAP API Tester window, or by right-clicking in the Project window and selecting Create > OSL > IAP > Setup Config.

Fields

Environment Environment Name production Unity Services environment name
Server URL https://your-server-domain.com Base URL of your backend verification server
Timeout Seconds 15 HTTP request timeout in seconds
Demo Defaults Default User ID TEST_USER_001 Default user ID used for demo and testing
Demo Consumable Product ID gold_100 Consumable product ID for the demo buy button

Note: If Server URL is left at its default value (https://your-server-domain.com), a warning is logged at runtime initialization.

Scene Setup

  1. Add an empty GameObject to your scene and attach the IapManager component.
  2. Assign your IapSetupConfig asset to the Setup Config slot.
  3. Assign a BackendApiClient component to the Api Client slot.
  4. Set the Current User ID field to the logged-in user's ID, or call SetCurrentUserId(userId) in code before any purchase is initiated.

Checking Setup Warnings

IapManager.GetSetupWarnings() returns an array of warning messages for any missing or invalid configuration. Review the Console output for these warnings before testing.

Purchase Flow

BuyProduct(productId)
    └─ StoreController.PurchaseProduct()
           └─ OnPurchasePending (PendingOrder)
                  └─ BuildPurchaseRequest() — parse receipt and branch by store
                  └─ VerifyWithRetryAsync() — send to backend (up to N retries)
                         └─ HandleVerificationResponse()
                                ├─ ShouldGrantReward → GrantItemToPlayer() → OnRewardGranted event
                                └─ ShouldConfirmPurchase → StoreController.ConfirmPurchase()

Initiating a Purchase

iapManager.BuyProduct("gold_100");

Subscribing to the Reward Event

iapManager.OnRewardGranted += OnItemGranted;

void OnItemGranted(string productId)
{
    Debug.Log($"Reward granted: {productId}");
}

Store Unity Receipt Store Value Backend Endpoint Value Sent
Google Play GooglePlay /api/iap/google/verify purchaseToken
Apple App Store AppleAppStore /api/iap/apple/verify TransactionId + Payload
Mac App Store MacAppStore /api/iap/apple/verify TransactionId + Payload

Backend Verification API

IapManager communicates with the backend using the following JSON models.

Request (PurchaseRequest)

{
  "UserId": "user-123",
  "ProductId": "gold_100",
  "TransactionId": "GPA.1234-5678-9012-34567",
  "PurchaseToken": "...",
  "IsSubscription": false,
  "IsSandbox": false
}

Response (PurchaseResponse)

{
  "IsSuccess": true,
  "ResultCode": "Verified",
  "Message": "",
  "TransactionId": "GPA.1234-5678-9012-34567",
  "ShouldConfirmPurchase": true,
  "ShouldGrantReward": true
}

ResultCode Values

Code Meaning
Verified Receipt verification succeeded
AlreadyProcessed Transaction has already been processed (duplicate prevention)
VerificationFailed Receipt verification failed

Retry Logic

  • On verification failure, IapManager retries up to the Max Verification Retries count configured in IapSetupConfig.
  • The delay between retries increases progressively: RetryDelaySeconds × attempt number.
  • Retries stop immediately when Verified, AlreadyProcessed, or VerificationFailed is received, or when the reward or confirmation flag is true.
  • Concurrent processing of the same TransactionId is prevented by an in-memory tracking set.

IAP API Tester (Editor Tool)

Open the editor tool via Tools > OSL > IAP API Tester to test your backend verification API without running a full build.

Field Description
Setup Config Shared configuration asset. Use Load Defaults From Config to populate fields automatically.
Server Base URL Base URL of your backend server
Test User ID User ID to use in test requests
Product ID Product ID to include in the request
Is Subscription? Whether the product is a subscription
Is Sandbox (Apple)? Whether to set the Apple sandbox flag
Receipt Data For Google: purchaseToken. For Apple: TransactionId.

Click Test Google Receipt or Test Apple Receipt to send a request to /api/iap/google/verify or /api/iap/apple/verify respectively. The server response appears in the Server Response Log at the bottom of the window.

Operational Notes

  • Keep product IDs in sync between store consoles and your IapSetupConfig product catalog.
  • Validate both sandbox and live purchase paths before releasing to production.
  • Enable Force Apple Sandbox only during development; disable it in release builds.
  • Current User ID must be set before any purchase is initiated — verification is aborted if it is empty.
  • The in-memory _grantedTransactionIds set prevents duplicate rewards within a session but is cleared on app restart. Ensure your backend handles AlreadyProcessed responses correctly.
